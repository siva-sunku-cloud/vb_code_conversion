System Architecture
Overall structure, pipeline topology, and component inventory

AI & LLM Integration Architecture
Provider abstraction, prompting strategy, self-correction loop, and translation memory

Provider Abstraction — LiteLLM

All agents call LiteLLM (litellm.completion()) rather than any vendor SDK. LiteLLM normalises the API across five providers. Switching providers requires only a .env change — no code changes anywhere in the pipeline.

BaseAgent — Shared LLM Utilities

agents/base_agent.py is an abstract base class (ABC). Every spoke agent extends it. It provides four utilities that all agents inherit:

Method
_call_llm()

Single LLM invocation via litellm.completion(). Signature: (system: str, messages: list[dict], max_tokens: int = 4096, tools: list | None) → ModelResponse. Prepends system prompt, handles Azure-specific parameters, logs token counts at DEBUG.

Method
_extract_code()

Strips markdown fenced code block (```python … ```) from an LLM response, returning only the inner code. Used by ConverterAgent to isolate the generated Python module from surrounding explanation prose.

Method
_parse_json()

Extracts text, strips markdown fences, calls json.loads(). Used by UnderstandAgent, ArchitectAgent, and ArchAuditor — all of which expect structured JSON responses.

Prompt Architecture — System + User Separation

Every agent call uses a two-part prompt structure. The split is a deliberate design choice: the system prompt defines constraints that never change; the user message carries context that varies per call and per retry.

PartContentChanges between calls?Design rationale
System promptAgent persona, output format requirements, hard constraints (e.g. "return only valid JSON with these keys", "no prose after the code block", "never use global mutable state")No — static constant in each agent fileConsistency: the same rules apply on every call and every retry. Testable in isolation.
User messageDynamic context: Java source code, analysis JSON, test file, error output from mypy/pytestYes — augmented with tool error output on retriesPrecision: the retry prompt contains exact compiler/test output — line numbers, assertion values — not a vague "it failed" signal.

Self-Correction Loop (Step 3)

The closed feedback loop is the central AI design pattern in the pipeline. It provides the Converter Agent with exact machine-generated error output as its next prompt's context — the same feedback cycle a human engineer uses. This is sometimes called tool-augmented reflection.

⚖️
Why precise feedback produces effective corrections
mypy errors include exact file path, line number, and specific type mismatch description. pytest failure traces include the exact assertion that failed and actual vs expected values. This precision means the model knows exactly which line to fix — not just "something is wrong." In practice, the majority of modules pass on attempt 1; most of the remainder pass on attempt 2.

Parallel Execution (Step 2)

The Functional Tests Agent and Golden Master Agent are stateless and produce independent outputs. The Hub Agent runs both concurrently via asyncio.gather(), reducing Step 2 wall-clock time by ~35% compared to sequential execution.

# Both LLM calls are in-flight simultaneously
# Total latency = max(t_functional, t_golden)  not  t_functional + t_golden
functional_suite, golden_suite = await asyncio.gather(
    FunctionalTestsAgent().run(spec, arch_path),
    GoldenMasterAgent().run(spec, io_samples_path),
)

Translation Memory — Qdrant Vector DB

Before generating Python code, the Converter Agent performs a cosine-similarity search for previously solved Java→Python patterns. This reduces redundant re-derivation of common translation constructs across modules.

OperationWhenWhat is stored / queried
Query (find_similar)Before each Converter generation attemptEmbedding of current architecture's design_notes + public_api signatures. Top-5 similar patterns retrieved and injected as examples into the system prompt.
Store (store_pattern)After Human Gate approval in Step 3Architecture summary + generated code digest. Available to all subsequent modules in the session (and across sessions with a persistent Qdrant instance).

Graceful degradation: if Qdrant is unavailable, the vector DB client falls back to in-memory mode (QdrantClient(":memory:")) automatically. Memory resets on process exit. The pipeline never errors due to Qdrant unavailability.

Hub Agent
agents/hub_agent.py — Orchestrator, state machine, and MCP server lifecycle manager

The Hub Agent is not an AI agent. It does not call any LLM. Its role is deterministic orchestration: it owns the global state machine, starts and stops the four MCP server subprocesses, dispatches spoke agents, and presents human review gates. All intelligence lives in the spoke agents.

Key Methods

MethodPurposeSpoke agents dispatched
run(request)Entry point. Starts all MCP servers in AsyncExitStack, scans for .java files, iterates per module.—
_analyze_and_design()Runs Steps 1-A → 1-B → 1-C sequentially, then both validators. Calls _human_gate() unconditionally.UnderstandAgent, DocumentAgent, ArchitectAgent, ASTChecker, ArchAuditor
_build_test_execute()Runs test agents in parallel via asyncio.gather(), then Test Auditor, then dry run. Calls _human_gate().FunctionalTestsAgent, GoldenMasterAgent, TestAuditor, DryRunRunner

MCP Server Lifecycle

# AsyncExitStack guarantees all 4 server subprocesses shut down cleanly
# on any exit path — normal completion, exception, or KeyboardInterrupt
async with AsyncExitStack() as stack:
    self.fs       = await stack.enter_async_context(FilesystemClient(output_dir))
    self.executor = await stack.enter_async_context(ExecutionClient(output_dir))
    self.vectordb = await stack.enter_async_context(VectorDBClient())
    self.github   = await stack.enter_async_context(GitHubClient())
    await self._run_pipeline(request)

State is persisted to migration_state.json after every status change via MigrationManager. Interrupted runs can be resumed at the last successful step using --start-step.

Analysis Agents — Step 1
agents/analyze_and_design/ — Three LLM agents and two validators; run sequentially
AgentInputOutput modelLLM?Key design decision
UnderstandAgent Java source (str) AnalysisResult Yes System prompt demands strict JSON — no prose. Output persisted to disk so --start-step 1b skips the LLM call on reruns, saving cost.
DocumentAgent AnalysisResult MarkdownSpec Yes Markdown chosen as output format — machine-parseable by the Test Auditor, human-readable by domain experts. JSON would be too low-level; free prose would be unstructured.
ArchitectAgent AnalysisResult + MarkdownSpec ArchitectureDesign Yes Receives analysis and spec but not Java source. Deliberately decoupled from Java — produces a Python-idiomatic blueprint, not a Java translation. Converter in Step 3 is then given this blueprint, not the Java source.
ASTChecker AnalysisResult + Java source ValidationResult No — regex Regex over a real Java AST parser (e.g. javalang) to avoid a JVM/parser dependency. Non-blocking: issues are warnings shown at the Human Gate, not pipeline halts.
ArchAuditor ArchitectureDesign ValidationResult Yes Separate from ArchitectAgent — an "author + reviewer" separation that reduces blind spots. The Auditor has a checklist-focused system prompt (global state? Java-isms? missing type hints?), not a design prompt.
⚖️
Why the Converter never sees the Java source code
Giving the Converter the Java source risks producing a line-by-line translation rather than idiomatic Python. By constraining it to the architecture blueprint and the test file, the model is forced to reason about the target language's idioms independently. The architecture enforces Python patterns (Protocols for DI, dataclasses, X | None) that the Converter then implements.

Test Agents — Step 2
agents/build_test_execute/ — Two parallel generators, one LLM auditor, one syntax validator
AgentInputOutputLLM?Key design decision
FunctionalTestsAgent MarkdownSpec TestSuite (functional) Yes Spec-to-test: every numbered business rule → at least one test. System prompt forbids mocking business logic — only external I/O (DB, HTTP) may be mocked.
GoldenMasterAgent MarkdownSpec + optional I/O data TestSuite (golden_master) Yes Two modes: (1) real I/O data → exact @pytest.mark.parametrize from historical Java outputs; (2) synthetic fallback → LLM infers expected outputs from spec. Real data mode is significantly more reliable.
TestAuditor Both TestSuite objects + MarkdownSpec Merged TestSuite Yes Separate reviewer agent — not part of either generator. Removes trivial assertions, eliminates over-mocking, fills spec-rule coverage gaps, resolves duplicate names. The merged file is the Step 3 acceptance contract.
DryRunRunner Merged test file path ExecResult No — MCP tool Uses py_compile (not pytest --collect-only). At this point the Python module doesn't exist yet, so imports would fail on --collect-only. Pure syntax validation is the correct scope here.
⚖️
Why three agents instead of one comprehensive test generator
The Functional Agent maximises spec-rule coverage; the Golden Master Agent maximises output fidelity; the Test Auditor specialises in quality and deduplication. Each has a tightly-scoped system prompt and objective. A single agent attempting all three objectives produces mediocre results in all three — a well-known failure mode of over-specified LLM tasks.

Converter Pipeline — Step 3
agents/generate_code/ — Code generation in a closed mypy → pytest correction loop
ComponentInputOutputLLM?Key design decision
ConverterAgent ArchitectureDesign + merged TestSuite + Qdrant top-5 examples ConversionResult Yes Never sees Java source (by design). Instruction "Fix ALL failing tests" on retries — the word ALL prevents the model from addressing only the first failure. Qdrant examples reduce re-derivation of known patterns.
MypyChecker Generated .py file path ExecResult No — MCP tool Runs mypy in-process via mypy.api.run() — not subprocess. Avoids Windows shell-pipe byte encoding issues. Runs before pytest: type errors cascade into multiple test failures; fixing them first reduces total retries needed.
PytestRunner Test file path ExecResult No — MCP tool Runs pytest as a subprocess (not in-process). Subprocess isolation prevents a bad import or sys.exit() in generated code from crashing the pipeline process. Full stdout+stderr returned for retry injection.

Data Models
models/artifacts.py — Pydantic v2 typed contracts flowing between every agent

All inter-agent data is exchanged via Pydantic models. Every agent's run() method accepts and returns typed objects — never raw dicts or untyped strings. Benefits: runtime validation at every boundary, full JSON serialisation for disk persistence (enabling --start-step resume), and explicit, auditable data contracts.

MCP Infrastructure
infrastructure/ — Four server/client pairs using MCP stdio transport for tool isolation

MCP (Model Context Protocol) is used here as a local IPC mechanism. Each tool is a Python subprocess communicating over stdin/stdout with JSON-RPC. This provides three structural benefits: replaceability (swap the storage backend without touching agents), process isolation (a crash in the execution server cannot crash the pipeline), and schema-defined contracts (JSON Schema per tool — the same contract an LLM could call directly).

📁
FileSystem Server/Client
All file I/O. Agents never call pathlib.open() directly — everything flows through FilesystemClient. Makes the storage backend (local disk, S3, etc.) swappable without touching agent code.
💻
Execution Server/Client
Sandboxed pytest (subprocess) and mypy (in-process via mypy.api). Returns structured ExecResult(returncode, stdout, stderr). The combined property concatenates both for direct prompt injection.
🗄️
VectorDB Server/Client
Qdrant translation memory. store_pattern() after each successful migration. find_similar() before each Converter generation. Auto-falls back to in-memory Qdrant if QDRANT_URL is unset.
🐙
GitHub Server/Client
Creates feature branch migration/{module} and opens a PR with test results in the description. Silently skips if GITHUB_TOKEN/GITHUB_REPO is absent — no error raised, no pipeline impact.

BaseMCPClient — Lifecycle Pattern

infrastructure/clients/base_client.py is an abstract async context manager. All four clients extend it. It manages subprocess spawning, MCP protocol handshake, tool invocation (with 180-second timeout), and clean shutdown — in that order on every async with block.

⚖️
Why MCP rather than direct function calls?
Direct function calls would tightly couple agent logic to infrastructure implementation. MCP's JSON Schema tool definitions make the interface explicit and versionable. Critically, the same MCP server could be called directly by an LLM using tool-use — enabling future architectures where the LLM itself calls tools, not just Python wrapper code.

TDD applied at the pipeline level
Step 2 runs in full before Step 3 begins — the test suite exists, has been human-approved, and has passed a syntax check before the Converter writes a single line of Python. The Converter is evaluated against this pre-existing contract, not a test suite it co-generates. This is test-driven development applied to AI code generation.

Key Design Decisions & Trade-offs
Architectural choices, the rationale behind them, and the trade-offs accepted
DecisionChoice madeWhyTrade-off accepted
LLM provider coupling LiteLLM abstraction — all providers via one call Prevents vendor lock-in; allows cost/capability optimisation per provider; enables A/B testing of providers on the same pipeline LiteLLM is a runtime dependency; provider-specific advanced features (e.g. Anthropic extended thinking, OpenAI structured outputs) require LiteLLM support
Infrastructure boundary MCP server/client pairs for all I/O and tool execution Process isolation, schema-defined contracts, swappable backends without agent changes Additional startup latency (subprocess spawn); more complex debug path when a tool call fails
Converter input scope Converter receives architecture + tests — never Java source Forces Python-idiomatic implementation; prevents literal Java translation The Converter cannot reference the original Java logic to resolve ambiguity; must rely entirely on the architecture and tests
Type safety enforcement mypy --strict as a mandatory gate before pytest Type errors cascade into multiple test failures; catching them first reduces total LLM retry cycles needed --strict is unforgiving — third-party libraries without type stubs require --ignore-missing-imports; complex generics can produce false positives
Test generation strategy Three separate agents (Functional + Golden Master + Auditor) instead of one Each agent has a tightly-scoped objective; the Auditor adds a quality pass neither generator would perform on its own Three LLM calls in Step 2 instead of one; though two of the three run in parallel, cost is higher than a single-agent approach
Human gates Mandatory y/n prompt after each step — no configurable threshold An AI pipeline migrating business-critical Java code requires explicit human accountability at each stage; no automated quality score is reliable enough to skip review Fully automated pipelines are not possible; a human must be present for each module. Acceptable for migration projects; not suitable for high-volume automated translation
Java parsing approach Regex patterns in ASTChecker instead of a Java AST parser Avoids a JVM or javalang dependency; sufficiently precise for structural completeness checking Regex cannot handle all Java constructs; inner classes, annotations, and generics may be misidentified. Non-blocking design mitigates the impact of false negatives.
Translation memory Qdrant vector search as a best-effort lookup, not required infrastructure Improves generation quality progressively over a migration project without making it a hard dependency In-memory mode resets on exit; no cross-session learning without a persistent Qdrant instance. The quality benefit is marginal on the first few modules.
