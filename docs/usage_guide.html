Usage & Operations Guide

Java → Python Migration Orchestrator

A multi-agent AI pipeline that converts Java classes and modules to idiomatic, type-annotated Python — with full test coverage — in three automated steps, each with a mandatory human review gate. This guide covers everything from setting up your first migration to reviewing each stage and troubleshooting failures.

Architecture & Design
How the system is structured and why

🎓 Stakeholders The orchestrator uses a Hub & Spoke pattern — one central Hub Agent coordinates a set of specialised agents (spokes), each responsible for a single concern such as understanding Java code, writing tests, or generating Python. No agent does more than one job.

🔧 Tech Team Every agent calls the Claude API (Anthropic) with a dedicated system prompt. The Hub Agent is the sole keeper of migration state; spoke agents are stateless and receive all context they need in their prompt.

AI & LLM Architecture
How the pipeline integrates with language models — provider abstraction, prompting strategy, and self-correction

🔧 Tech Team 🏛️ Architects Every spoke agent is an AI agent — it formulates a structured prompt, invokes an LLM, and parses a typed response back into a Pydantic model. This section explains the AI layer end-to-end: how the provider abstraction works, how prompts are structured, how the self-correction feedback loop operates, and how parallel execution reduces latency.

LiteLLM — Provider Abstraction Layer

All agents call LiteLLM (litellm.completion()) rather than any vendor SDK directly. LiteLLM normalises the completion API across providers — the same Python call works against Anthropic Claude, OpenAI GPT-4o, Google Gemini, GitHub Copilot, and Azure OpenAI. The active provider is selected at startup from LLM_PROVIDER in .env and injected by the Config class.

File: agents/base_agent.py — Abstract base class (ABC). Every spoke agent extends this. Never instantiated directly. Provides four shared utilities:

Prompt Anatomy — System + User Messages

Every agent call follows the same two-part prompt structure, consistent across all LLM providers:

Part 1
System Prompt
static constant in agent file

Defines the agent's persona, output format, and hard constraints. For the Understand Agent, the system prompt specifies that the response must be valid JSON with exactly these top-level keys: business_logic, data_structures, external_api_calls, control_flow, complexity_score, raw_summary. No prose outside the JSON block is permitted.

System prompts are module-level string constants — they never change between invocations. Only the user message varies per call.

Part 2
User Message
built dynamically per call

Contains the variable context for this specific invocation — Java source code, analysis JSON, test failure output, etc.

On retry calls in Step 3, the user message is augmented with the complete tool error output. For example:

"mypy found 2 errors — fix them before writing the full implementation: [full mypy output here]"

This is how the Converter Agent performs self-correction without any persistent memory between calls.

Self-Correction Loop — The Core AI Pattern

The closed feedback loop in Step 3 is the most architecturally significant AI design in the system. Rather than asking an LLM to "review its own code," the loop feeds exact, machine-generated compiler and test output back as the next prompt's context. 🔬
Why precise feedback matters
mypy errors include exact file path, line number, and type mismatch description. pytest failure traces include the assertion that failed and the actual vs expected values. This precision makes corrections highly targeted — the model knows exactly which line to fix, not just "something is wrong." In practice, ~70% of modules pass on the first attempt; most of the rest pass on retry 1 or 2.
💰
Token cost of retries
Each retry sends the full generated code (typically 100–400 lines) plus the error output back to the LLM. For a complex module, one retry can cost 2,000–8,000 output tokens. Set MAX_RETRIES=5 for very complex modules, but monitor spend. The DEBUG log file records exact token counts per call.

Parallel Execution (Step 2)

The Functional Tests Agent and Golden Master Agent are stateless and independent of each other. The Hub Agent exploits this with asyncio.gather() — both LLM calls are in-flight simultaneously: 

# Both LLM calls are in-flight at the same time
# Total latency = max(t_functional, t_golden) instead of t_functional + t_golden
functional_suite, golden_suite = await asyncio.gather(
    FunctionalTestsAgent().run(spec, arch_path),
    GoldenMasterAgent().run(spec, io_samples_path),
)

For a typical module, Functional Tests takes ~15–25 s and Golden Master takes ~10–20 s. Parallel execution reduces Step 2 wall-clock time by ~35% compared to sequential calls.

Qdrant Translation Memory

Before generating Python, the Converter Agent performs a semantic search for previously solved Java→Python patterns in Qdrant. This reduces redundant re-derivation of common translation patterns.

🔍
Query at generation time
The Converter encodes the current architecture's design_notes + public_api signatures. Top-5 most similar past migrations are retrieved and injected as examples into the system prompt.
💾
Store on success
After Human Gate approval in Step 3, the successful Java→Python pattern (architecture summary + code digest) is stored in Qdrant. Available to all subsequent modules in the session.
📈
Improves over time
With a persistent Qdrant instance, translation memory accumulates across sessions. The 50th module benefits from 49 prior solved examples. In-memory mode resets on exit.
⚠️
Graceful degradation
If Qdrant is unavailable (no Docker, no QDRANT_URL), the Converter skips the lookup and generates from scratch. Pipeline continues without errors — memory is best-effort, not required.

Step 1 — Discovery, Documentation & Architecture
Understanding the Java module and designing the Python equivalent

🎓 Stakeholders Step 1 answers the question "What does this Java code actually do?" It produces a plain-English Markdown document you can read and a machine-readable architecture blueprint for the Python replacement. A mandatory human review gate always triggers at the end of Step 1.

A
Understand Agent
agents/analyze_and_design/understand_agent.py · Model: claude-sonnet-4-6

Reads the Java source and produces a structured JSON analysis covering all classes, methods, external API calls, control flow (for/while/try-catch), error handlers, and a complexity score (0–1).

AnalysisResult JSON complexity_score: 0.0–1.0

What to check

  • 🔍
    All class and method definitions captured
    The AST Checker (see below) flags any methods found in source but missing from the analysis.
  • 🔍
    Complexity score is reasonable
    0.0 = trivial CRUD. 1.0 = deeply nested logic with multiple checked exceptions and external calls. The human gate always triggers after Step 1 completes.
  • 🔍
    External API calls listed
    Every external dependency (JDBC, JPA, REST clients) must be identified here — these become dependency-injection points in the architecture.
B
Document Agent
agents/analyze_and_design/document_agent.py · Model: claude-sonnet-4-6

Converts the JSON analysis into a Markdown specification covering overview, inputs/outputs table, numbered business rules, data structure tables, error conditions, and migration notes for Java-specific idioms.

{module}_spec.md

What to check

  • 📋
    Business rules are accurate
    Have a domain expert read the numbered list. These rules are the source-of-truth for test generation in Step 2.
  • 📋
    All Java idioms noted
    Null returns, checked exceptions, synchronized blocks, and raw collection types must appear in "Migration Notes".
  • 📋
    Inputs/outputs table complete
    Every public method must have a row showing parameters, types, and return values.

Architect Agent
agents/analyze_and_design/architect_agent.py · Model: claude-sonnet-4-6

Designs the Python equivalent: Pydantic/dataclass models, fully-annotated public API signatures, module boundaries, and dependency injection points for all external services.

{module}_architecture.json

What to check

  • 🏛
    No global mutable state
    The Arch Auditor flags this automatically. All shared data must flow through function parameters or injected services.

What to check in the merged test file

  • 🧪
    At least one test per business rule
    Cross-reference against the numbered list in the Markdown spec. Every rule should map to ≥ 1 test.

  • 🧪

Quality Checklist
What to review at each stage before signing off

🔧 Tech Team Use this checklist when manually reviewing output artifacts or when a module fails the pipeline and you're diagnosing the cause.

Step 1 — Analysis & Architecture

  • ☐
    complexity_score is proportionate
    Simple utilities should score < 0.3. Modules with deeply nested loops, multiple checked exceptions, and external API calls should score > 0.6.
  • ☐
    All public methods in analysis.control_flow.methods
    Open the Java file and count public method declarations. Counts must match the analysis output.
  • ☐
    Markdown spec matches actual Java behaviour
    Have a developer familiar with the original system read section 2 (Business Rules). Incorrect rules here cascade into wrong tests and wrong code.
  • ☐
    Architecture has zero global mutable state
    Search the architecture JSON for "global" or "module-level" in design_notes. Any mutable dict/list at module scope is a red flag.
  • ☐
    Every JDBC/JPA/REST dependency is an injected abstraction
    e.g. JdbcTemplate or RestTemplate → protocol class injected into the constructor, listed in dependency_injection.
  • ☐
    No Java getter/setter naming in Python
    getCustomerName() → customer_name property. setCount(n) → direct attribute assignment.

Step 2 — Test Suite

  • ☐
    One test per business rule minimum
    Count numbered rules in the spec. Count def test_ functions covering each. Gaps must be added manually.
  • ☐
    Boundary values tested
    For ApplyDiscount: 0%, 30% (max), 31% (over), and negative inputs. All four should have test cases.
  • ☐
    Exception paths tested
    Every business rule that says "raises X on Y" must have a with pytest.raises(X) test.
  • ☐
    No assert True, assert 1 == 1
    Grep the test file: grep "assert True\|assert 1 == 1" test_*.py — output must be empty.
  • ☐
    Golden master data matches actual Java output
    If you have real Java I/O logs, spot-check 5 parametrize cases against them.
  • ☐
    Dry run passes (all tests collected)
    Run manually: pytest --collect-only OUT/{module}/test_{module}.py

Step 3 — Generated Python Code

  • ☐
    mypy --strict exits 0
    Run: mypy --strict OUT/{module}/{module}.py
  • ☐
    All tests pass
    Run: pytest -v OUT/{module}/test_{module}.py (where OUT = your output dir)
  • ☐
    Line count is proportionate
    Python should be ~50–70% of the Java line count (less ceremony, same logic). Suspiciously short = likely missing logic.
  • ☐
    No Any types in production paths
    Run: grep "Any" OUT/{module}/{module}.py — any hits should be justified with a comment.
  • ☐
    Error handling is specific
    No bare except: clauses. Each exception should name the specific type (e.g. except ValueError).
  • ☐
    Dependencies are injected (not hardcoded)
    No hardcoded connection strings, file paths, or API endpoints in the module body.

Adding Your Own Tests

Add tests directly to OUT/{module}/test_{module}.py. Follow these conventions to keep them compatible with the pipeline:

  • ✏️
    Use pytest.mark.parametrize for multiple input variants
    Avoids copy-paste test functions that all look the same.
  • ✏️
    Write a one-line docstring referencing the spec rule number
    e.g. """Rule 4: VAT is applied after discount."""
  • ✏️
    Use pytest.raises(ExceptionType, match="pattern")
    The match pattern ensures the right exception message is checked, not just the type.
  • ✏️
    Use Decimal for all monetary values
    Floating-point rounding errors will cause flaky golden master tests. Always use Decimal("0.10") not 0.10.

MypyChecker — mypy_checker.py

CHK
MypyChecker
agents/generate_code/mypy_checker.py · no LLM · uses ExecutionClient (MCP)

Role: Calls the Execution MCP server's run_mypy tool, which invokes mypy via its Python API (mypy.api.run()) — not as a subprocess. Running in-process eliminates shell-pipe encoding issues that are common when spawning mypy as a subprocess on Windows (byte encoding of error messages).

Flags used: --strict --ignore-missing-imports --follow-imports=skip. --strict enforces: no implicit Any, return types on all functions, None-safety checks. --ignore-missing-imports prevents failures due to third-party stubs not being installed. --follow-imports=skip prevents mypy from recursing into the project's other modules.

Why run mypy before pytest? Type errors cause multiple test failures — a wrong return type on one function can cascade into 5–10 test failures. Running mypy first provides a more targeted error signal to the Converter; it knows exactly which type annotations are wrong before wading through test output. This reduces the number of retry cycles needed.

Timeout handling: The ExecutionClient.run_mypy() method wraps the MCP call in a 180-second asyncio.wait_for(). On timeout, it returns a synthetic ExecResult(returncode=-1, stderr="mypy timed out"), which is treated as a mypy failure and triggers a Converter retry.

MCP Protocol — What It Is and Why It's Used Here

MCP (Model Context Protocol) is an open protocol for connecting AI agents to tools via a standardised JSON-RPC interface. In this pipeline, it is used as a local IPC mechanism: each server is a Python subprocess communicating over stdin/stdout. This design provides three key benefits for an agentic pipeline:

🔄
Replaceability
The same MCP interface allows the execution backend to change without touching agent code. The filesystem server could point to S3; the execution server could run in a Docker container.
🛡️
Isolation
Running tools as separate subprocesses means a crash in the execution server (e.g. bad test code triggers a segfault) cannot crash the pipeline process. The MCP connection simply times out or disconnects.

BaseMCPClient — infrastructure/clients/base_client.py

Abstract base that every MCP client extends. Manages the full subprocess lifecycle as an async context manager:

  • 1️⃣
    Spawn subprocess
    Uses StdioServerParameters(command=sys.executable, args=[server_script] + server_args). Sets PYTHONPATH to the project root in the subprocess environment so the server can import from config, utils, etc. even when launched from a different working directory.
  • 2️⃣
    Protocol handshake
    Creates a ClientSession and calls session.initialize() — the MCP capability negotiation that registers available tools and their schemas.
  • 3️⃣
    Expose _call(tool, **kwargs)
    Sends a call_tool JSON-RPC request. Wrapped in asyncio.wait_for(timeout=180.0) to prevent agent hangs. Returns the first text content block from the response as a string.
  • 4️⃣
    Clean shutdown
    __aexit__ closes the ClientSession and the stdio transport, which sends a graceful terminate signal to the subprocess. The AsyncExitStack in the Hub Agent calls this in LIFO order for all four clients on any exit path.

Re-running a Failed Module

Use --start-step to resume from the last successful step instead of re-running the whole pipeline. The run.sh interactive wrapper reads migration_state.json and suggests the correct step automatically.

# Recommended: use run.sh — it reads migration_state.json and offers a resume menu
./run.sh your_java_project/

# Manual: skip Steps 1 and 2 (artifacts already on disk), re-run only Step 3
python main.py your_java_project/ --start-step 3 --retries 5

# Resume at Step 1-C (reload analysis + spec from disk, re-run architect only)
python main.py your_java_project/ --start-step 1c

# Fallback: isolate a single file in a temp dir if state is badly corrupted
mkdir tmp_retry && cp your_java_project/InvoiceCalculator.java tmp_retry/
python main.py tmp_retry/ --retries 5
rm -rf tmp_retry

Making the Tool Interactive
Options for adding a UI, step-by-step wizard, or dashboard to the pipeline

The pipeline currently runs as a headless CLI. Three approaches add progressively richer interactivity — choose based on your team's comfort and the project's scope.

Option A — Streamlit Dashboard (Recommended for teams)

A Streamlit app can drive the Hub Agent step-by-step, showing artifacts in real time and collecting human-gate decisions through UI forms instead of terminal prompts.

📊
Module Status Board
Live grid showing each Java file → status (Analyzing / Documented / Converting / Done / Failed). Colour-coded, refreshes every 2 s via st.rerun().
👁️
Inline Spec & Arch Viewer
Click a module row to open its spec.md and architecture.json side-by-side. Renders Markdown natively with st.markdown().
⚠️
Human Gate Dialog
Replace rich.Confirm.ask() with a st.dialog popup. The Hub Agent blocks on an asyncio.Event until the reviewer clicks Approve or Reject.
📝
Editable Spec & Tests
Render spec.md and test file in st.text_area() with a Save button. Write changes back via the File System MCP before proceeding.

Option B — Step-by-Step CLI Wizard

Keep the terminal but make each stage a confirmed step. Replace the automated for module_state in modules loop with an interactive prompt that shows the artifact before continuing.

💡
How to implement it
In agents/hub_agent.py, after each await self.fs.write_text(spec_path, spec.content), add a rich.Confirm.ask(f"Step 1-B done. Open {spec_path} to review. Continue to Step 1-C?"). This turns the pipeline into a step-by-step wizard with a single-line change per stage.

Option C — Jupyter Notebook Driver

For exploratory or one-off migrations, a notebook gives cell-by-cell control with full Python flexibility.
